# UI Audit — Claude Opus 4.8

## Verdict
NEEDS-WORK — The app is theme-clean and semantically correct, but it has **no mobile layout at all**: a fixed 288px sidebar permanently steals ~77% of an iPhone 13 mini's width, several core actions (delete session) are physically unreachable on touch, and there's no safe-area / PWA / viewport-fit handling — so it's usable on a Framework workstation but broken as a first-class iPhone tool.

## Critical Issues (mobile-breaks-the-app)

1. **`src/components/Sidebar.tsx:40` + `src/App.tsx:24` — fixed `w-72` sidebar always rendered beside chat; no drawer/collapse.** `App` lays out `<Sidebar>` and `<Chat>` as flex siblings with the aside at `w-72 shrink-0` (288px). At 375px that leaves **87px** for the entire chat column (message list + composer); at 390px → 102px; at 430px → 142px. The chat is effectively unusable on every target iPhone width. There is **zero** responsive logic — confirmed by grepping `src/components/*.tsx`: the only `sm:`/`md:` utilities in the codebase live inside `src/components/ui/` primitives, never in app layout. *Fix: make the sidebar a slide-over on small screens using the existing Radix Dialog/`Sheet` pattern (shadcn `sheet` is the canonical "Dialog as side drawer"), hidden behind a hamburger; render it inline only at `md:` and up (`hidden md:flex` on the aside, a trigger button in a mobile top bar). Reuses existing Radix portal + Dialog machinery — no new component class.* Confidence: **high**.

2. **`src/components/Sidebar.tsx:68` — session delete (`×`) is unreachable on touch.** The delete button is `className="hidden … group-hover:block"`, and Tailwind v4 compiles `group-hover:` inside `@media (hover:hover)` (verified in the shipped CSS: the single `@media (hover:hover)` block wraps exactly this `.group-hover\:block` rule). iPhones report `hover: none`, so the button **never becomes visible** and there is no alternative delete path (grep shows `remove()` is wired only to this one button; no swipe, no long-press, no overflow menu). Sessions can be created on mobile but never deleted. *Fix: drop the hover-gating — make the control always visible on touch (e.g. `block md:hidden md:group-hover:block`, or just always-visible with reduced opacity), or move delete into a per-row menu / swipe action. Simplest: render the `×` unconditionally and bump its hit area.* Confidence: **high**.

3. **`index.html:6` — viewport meta lacks `viewport-fit=cover`; combined with zero safe-area CSS, content collides with the notch and home indicator.** Current tag: `width=device-width, initial-scale=1.0`. The compiled CSS contains **0** `env(safe-area-inset-*)` references (verified against `dist/assets/*.css`). The chat composer (`Chat.tsx:55`, a bottom `border-t` bar) and the sidebar bottom will sit under the home-indicator bar; status-bar area is unpadded. *Fix: add `viewport-fit=cover` to the meta, then pad the composer and any full-height edges with `pb-[env(safe-area-inset-bottom)]` / `pt-[env(safe-area-inset-top)]` (Tailwind v4 supports arbitrary `env()` values directly). This is the standard, no-new-component approach.* Confidence: **high**.

## High Priority (degrades mobile UX significantly)

1. **`src/components/ui/button.tsx:25` — default button height is `h-8` (32px); icon button is `size-8`.** Every actionable control in the app uses the default size — Send/Stop (`Chat.tsx:67/72`), New chat / Tasks / ⚙ (`Sidebar.tsx:42/45/48`), task Cancel/View (`TaskCard.tsx:115/118` use `size="sm"` → `h-7`/28px, even smaller), schedule Cancel (`Settings.tsx:111`). All are well under Apple HIG's 44pt minimum; the `sm` (28px) and `icon-sm` (28px) variants used in cards/dialogs are worse. *Fix: don't fork the primitive — add a touch-sized treatment at the call sites or globally bump the default. Cleanest harness-friendly option: in `index.css` add a coarse-pointer rule that grows controls, e.g. `@media (pointer: coarse) { [data-slot="button"], [data-slot="input"], [data-slot="select-trigger"] { min-height: 44px; } }`. One CSS block, no component rewrites, respects the "use existing CSS/Tailwind patterns" rule.* Confidence: **high** (sizes read directly from CVA variants).

2. **`src/components/Settings.tsx:56`, `TasksDialog.tsx:20`, `TaskCard.tsx:69` — dialogs are centered narrow boxes, not full-screen, on mobile.** Each passes `max-w-2xl`/`max-w-3xl`. Via `tailwind-merge` these resolve to `w-full sm:max-w-sm max-w-2xl` (verified) — the unprefixed `max-w-2xl` (42rem) overrides `sm:max-w-sm` at all widths by source order, so on a 375px screen the dialog is `w-full` minus the `-translate-x-1/2` centering, but it's vertically centered with `top-1/2 -translate-y-1/2` and only `max-h-[80vh]`. With the iOS keyboard open (Settings persona textarea, `rows={12}`), an 80vh centered box gets crushed and the Save button can end up under the keyboard. Settings has no `max-h`/scroll at all (`Settings.tsx:56`), so a long persona + schedules list can overflow the viewport with no internal scroll. *Fix: give dialogs a mobile-full-screen treatment — e.g. `className="max-w-2xl h-[100dvh] max-h-[100dvh] rounded-none sm:h-auto sm:max-h-[80vh] sm:rounded-xl"` and add `overflow-y-auto` to Settings' content. Use `dvh` not `vh` (see below). Standard shadcn responsive-dialog pattern.* Confidence: **medium** (cascade verified; keyboard-occlusion is inferred, needs device confirmation).

3. **`src/App.tsx:24` & `src/components/Login.tsx:21` — `h-screen` (100vh) instead of `100dvh`.** iOS Safari's `100vh` includes the area under the dynamic toolbar, so the bottom of a `h-screen` flex column (the chat composer) is pushed below the visible viewport and hidden behind Safari's bottom bar until the user scrolls. *Fix: replace `h-screen` with `h-dvh` (Tailwind v4 ships `h-dvh`). Two-line change.* Confidence: **high** (well-documented iOS behavior; `h-screen` confirmed in both files).

4. **`src/components/Chat.tsx:27-29` — `scrollIntoView({behavior:"smooth"})` on every message tick fights the iOS keyboard.** On focus, iOS scrolls the focused composer into view; a competing smooth auto-scroll to `bottomRef` on each `streaming`/`messages.length` change can cause janky scroll wars and the input jumping while typing. *Fix: scroll the message container's `scrollTop` rather than `scrollIntoView` on the whole page, or gate auto-scroll to "only if already near bottom." Low-risk improvement; verify on device.* Confidence: **medium**.

## Medium Priority (rough edges)

1. **`index.html:8` — generic `<title>web</title>` and no app metadata.** Shows as "web" on the iOS tab/switcher and as the default home-screen label. *Fix: set `<title>ytsejam</title>`.* Confidence: **high**.

2. **No PWA / add-to-home-screen support anywhere.** Repo-wide grep for `manifest`, `apple-touch-icon`, `apple-mobile-web-app-capable`, `theme-color`, `standalone` returns nothing; `vite-plugin-pwa` is not in `package.json`. Saved to the home screen, the app launches in a plain Safari chrome with a default icon and a white/again-generic status bar. For a tool Brian opens "throughout the day," an installable icon with a proper splash/status bar is high-value. *Fix (harness-friendly, minimal): add a small `manifest.webmanifest` (name, theme-color matching `--background` dark `oklch(0.155 …)` ≈ `#0b1418`, `display: standalone`, the existing `favicon.svg` as a maskable icon) plus `<link rel="manifest">`, `<meta name="theme-color">`, `<meta name="apple-mobile-web-app-capable" content="yes">`, `<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">`, and an `apple-touch-icon`. No service worker needed for install — keep it light, don't pull in Workbox.* Confidence: **high**.

3. **`src/components/Login.tsx:23` — fixed `w-80` card.** 320px on a 375px screen is fine but tight with no horizontal padding guarantee on the smallest target; on very narrow widths it can touch edges. *Fix: `w-80 max-w-[calc(100%-2rem)]` (or `w-full max-w-sm` with `px-4` on the parent).* Confidence: **medium**.

4. **`src/components/Message.tsx:74` — assistant/user bubbles `max-w-[80%]`; long unbroken tokens overflow.** Code spans, long URLs, and the `<pre>` blocks in `ToolCallCard` (`Message.tsx:43` uses `whitespace-pre-wrap` + `overflow-x-auto`, good) but inline markdown code and bare URLs in prose have no wrap guard. The bubble itself is fine, but `prose` inline `<code>` and long links can force horizontal scroll inside the narrow chat column. *Fix: add `break-words`/`overflow-wrap-anywhere` to the prose container (`Message.tsx:86`) and `min-w-0` is already present on card children elsewhere — apply the same to the bubble's text wrapper. Tailwind `[overflow-wrap:anywhere]` or `break-words` on the prose div.* Confidence: **medium** (overflow is likely but exact trigger needs a render).

5. **`src/components/Message.tsx:86` — `prose-invert` hardcoded.** Markdown is always styled for a dark background. The app is currently always-dark (so it renders correctly today), but this contradicts CLAUDE.md's "never assume the dark theme" rule and will break if a light theme is ever enabled. *Fix: drop `prose-invert` and rely on `prose` + the `.dark` variant, or use `dark:prose-invert`.* Confidence: **high** (string read directly; impact is latent, hence medium priority).

## iPhone-Specific Concerns

- **Sticky `:hover` — mostly already handled, but with a footgun.** Tailwind v4 wraps `hover:`/`group-hover:` in `@media (hover:hover)` (confirmed in shipped CSS), so hover backgrounds won't "stick" on tap. The downside is the inverse: anything that *only* appears on hover (the Sidebar delete `×`, Critical #2) is invisible on touch. Audit hover-only affordances, not hover-color states.
- **Input zoom is actually avoided — keep it that way.** `input.tsx:11` and `textarea.tsx:10` use `text-base` (16px) with `md:text-sm`, i.e. 16px on phones (<768px) dropping to 14px only on desktop. iOS won't auto-zoom on focus. The native `<select>` in `Settings.tsx:73` uses `text-sm` (14px) and *could* trigger zoom on focus — bump to `text-base` or add `md:text-sm`.
- **Native controls are a win on iOS.** Settings uses a real `<select>` (`Settings.tsx:73`) → native iOS wheel picker, and `confirm()` for delete (`Sidebar.tsx:34`) → native sheet. Both are good mobile choices; keep them.
- **`field-sizing-content` textarea auto-grow is Safari 17.4+ only.** `textarea.tsx:10` relies on `field-sizing-content` for growth; older iOS Safari ignores it and falls back to `rows={2}`/`min-h-16`. Acceptable, but the composer won't grow on iOS < 17.4 — minor.
- **Safe area + composer.** The bottom composer bar (`Chat.tsx:55`) and `h-screen` together mean the Send/Stop button sits in or under the home-indicator zone on notched phones — this is the single most thumb-relevant gap (ties Critical #3 + High #3).
- **Theme-color / status bar.** With no `theme-color` meta, the iOS status bar and Safari chrome won't match the dark `--background`; expect a jarring light bar above the dark app.

## What I Could NOT Verify Without a Browser

- Exact pixel overflow / horizontal-scroll at 375/390/430 — I can prove the *math* (288px sidebar vs 375px viewport) and the absence of breakpoints, but the precise clipped layout, whether any element forces a horizontal scrollbar, and where text wraps need a rendered viewport.
- Whether the iOS keyboard actually occludes the Settings Save button / chat composer (High #2, #3) — inferred from `top-1/2` centering + `max-h-[80vh]` + `h-screen`, but keyboard geometry is device-specific.
- The `scrollIntoView` vs keyboard scroll-war (High #4) — plausible from the code but only observable live.
- Real hit-target ergonomics — I can read heights (`h-8`=32px, `h-7`=28px) and compare to 44pt, but tap success/mis-tap rate needs a device.
- `field-sizing-content` fallback behavior on a specific iOS Safari version.
- Whether `confirm()`/native `<select>` render correctly inside the dark `color-scheme` on a real iPhone (CLAUDE.md notes headless Chrome lies about this).

## Strengths

- **Theme discipline is excellent.** Every app component uses semantic tokens (`bg-background`, `text-muted-foreground`, `bg-sidebar-accent`, etc.); no raw palette classes — and `color-scheme` is correctly synced in `:root`/`.dark` (`index.css`), so native chrome paints dark. This is enforced by `test/theme.test.mjs`.
- **Inputs are already iOS-zoom-safe** (16px on mobile via `text-base` + `md:text-sm`) — a thing many shadcn apps get wrong.
- **`min-w-0 flex-1` + `truncate` is used correctly** on the flex children that need it (`Sidebar.tsx:62`, `TaskCard.tsx:103`, `Settings.tsx:100`), preventing flex-overflow in those rows.
- **Hover-color stickiness is non-issue** thanks to Tailwind v4's automatic `@media (hover:hover)` wrapping.
- **Native-first controls** (`<select>`, `confirm()`) are good mobile ergonomics and match the "harness, not re-implementation" principle.
- **`<pre>` tool-call output already handles overflow** (`overflow-x-auto`/`overflow-auto` + `whitespace-pre-wrap`, `Message.tsx:43/47`).

## Suggested Next Steps

1. **Make the sidebar a drawer below `md:`** (Critical #1) — wrap the existing `<aside>` in shadcn `sheet` (Radix Dialog side variant), add a hamburger to a slim mobile top bar, `hidden md:flex` the inline aside. This single change unblocks all chat usage on iPhone.
2. **Fix viewport + safe areas + dvh together** (Critical #3, High #3): add `viewport-fit=cover` to `index.html`, swap `h-screen`→`h-dvh` in `App.tsx`/`Login.tsx`, pad the composer with `pb-[env(safe-area-inset-bottom)]`. Small, high-leverage diff.
3. **Touch-target pass** (High #1): add one `@media (pointer: coarse)` block in `index.css` giving `min-height:44px` to `[data-slot="button"]`/`input`/`select-trigger`, and un-hide the Sidebar delete button on touch (Critical #2).
4. **Mobile-full-screen dialogs** (High #2): add the responsive `h-[100dvh] rounded-none sm:h-auto sm:rounded-xl` treatment to Settings/TasksDialog/transcript, and give Settings content `overflow-y-auto`.
5. **Lightweight PWA polish** (Medium #2): real `<title>`, `manifest.webmanifest`, `theme-color`, apple-mobile-web-app + apple-touch-icon meta — no service worker. Makes the home-screen install feel first-class for daily check-ins.
