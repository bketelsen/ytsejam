# Desktop UI Audit — Claude Opus 4.8

## Verdict
NEEDS-WORK — The post-mobile-sweep state is genuinely good on desktop (the `max-w-4xl` chat constraint, permanent sidebar, and dialog scroll-containers all land well), but Brian's flag is real: dialog content is constrained only by accident. The `DialogContent` grid items have `min-width: auto`, so any non-truncating, non-wrapping child (notably **markdown tables**, which the typography plugin renders with no overflow wrapper) can push a dialog — or the chat column — wider than its declared `max-w`. Plus a handful of mouse/keyboard ergonomics gaps (coarse-pointer 44px sizing bleeding onto desktop, no Cmd/Ctrl+Enter, sidebar delete hover-gated even on desktop in a confusing way).

## Critical Issues (broken or significantly degraded desktop UX)

1. **`src/components/ui/dialog.tsx:62` — `DialogContent` is `grid` with no `min-w-0` on its content track; content can defeat the declared `max-w`.** The content class is `grid w-full max-w-[calc(100%-2rem)] … sm:max-w-sm`, overridden at the call sites to `max-w-2xl` (Settings, Tasks) / `max-w-3xl` (transcript). The single child wrapping the body (`<div className="space-y-4">` in Settings:60, `space-y-2` in TasksDialog:24, `space-y-3` in TaskCard:79) is a **grid item, and grid items default to `min-width: auto`** — they refuse to shrink below their min-content width. Today this is masked because every *current* child is independently guarded (textarea has `w-full min-w-0`, `<select>` is `w-full`, schedule/task rows use `min-w-0 flex-1` + `truncate`, Message bubbles have `break-words [overflow-wrap:anywhere]`, and prose `<pre>` compiles to `overflow-x:auto` — all verified). The moment any child *isn't* guarded, the dialog grows past its `max-w`. That's exactly Brian's "dialogs not constraining their content." *Fix (harness-friendly, one line): add `[&>*]:min-w-0` to the `DialogContent` base class in `ui/dialog.tsx:62`, or add `min-w-0` to each content wrapper div. The grid-item `min-width:auto` trap is the structural root; closing it makes the `max-w` authoritative regardless of child content.* Confidence: **high** (grid is confirmed in source and compiled CSS; `min-width:auto` on grid items is spec default; no `min-w-0` exists on any of the three wrappers).

2. **`src/components/Message.tsx:84-88` — markdown tables in `prose` have no overflow container, so a wide table blows out both the transcript dialog (`max-w-3xl`) and the chat column (`max-w-4xl`).** I verified the compiled typography CSS: `.prose table` is `table-layout:auto; width:100%` with **no** `display:block; overflow-x:auto` wrapper (unlike `.prose pre`, which *does* get `overflow-x:auto`). `table-layout:auto` sizes to content, so a markdown table with many columns or long unbreakable cell text expands past `width:100%`, and because the prose div is a grid item with `min-width:auto` (see #1), it drags the whole dialog/column wider — producing a horizontal scrollbar on the page (chat) or an over-wide modal. Assistant messages with markdown tables are common (LLM output loves tables). *Fix: wrap prose tables in a scroll container. Cleanest Tailwind-only option is an arbitrary-variant on the existing prose div (`Message.tsx:86`): add `[&_table]:block [&_table]:overflow-x-auto [&_table]:w-max [&_table]:max-w-full` — or simpler, give the prose container itself `overflow-x-auto` so the table scrolls inside the bubble. Pair with #1's `min-w-0`.* Confidence: **high** (typography table CSS read from compiled output; absence of overflow wrapper confirmed).

## High Priority (degrades desktop ergonomics)

1. **`src/index.css:150-156` — the `@media (pointer: coarse)` 44px min-height rule fires on touchscreen *laptops/desktops*, inflating every button/input on hybrid devices.** The mobile sweep added `@media (pointer: coarse) { [data-slot="button"], [data-slot="input"], [data-slot="select-trigger"] { min-height: 44px; } }`. The trap: a Surface, a touchscreen XPS, or any 2-in-1 in laptop mode reports `pointer: coarse` as its *primary* pointer even with a mouse and a 1920px screen — so the entire UI (which is designed around the dense `h-8`/32px button) jumps to 44px controls at desktop width. This is the inverse of the mobile concern and a real desktop-density regression on a non-trivial hardware class. *Fix: tighten the query so it only applies where there's no fine pointer — `@media (pointer: coarse) and (hover: none)` — which excludes mouse-equipped touch laptops while still catching phones/tablets. One-token change, still pure CSS, no component edits.* Confidence: **high** (rule read directly; `pointer: coarse` primary-pointer semantics on hybrid devices are well-documented).

2. **`src/components/Chat.tsx:77-82` — Enter sends, Shift+Enter newlines, but there is no Cmd/Ctrl+Enter send and no visible hint; this is a desktop power-user expectation gap.** The composer is the single most-used desktop control. Enter-to-send is fine, but two things hurt keyboard-driven desktop use: (a) many users expect Cmd/Ctrl+Enter (Slack/Linear/GitHub muscle memory) and will be surprised either way without a hint; (b) `e.key === "Enter" && !e.shiftKey` also fires during IME composition (CJK input), sending half-composed text. *Fix: guard with `!e.nativeEvent.isComposing` before the Enter-send branch (one condition), and optionally also accept `(e.metaKey || e.ctrlKey) && e.key === "Enter"` as an explicit send. The IME guard is the higher-value half and is unambiguously correct.* Confidence: **high** for the IME issue (logic read directly); **medium** that Cmd+Enter is wanted (product call).

3. **`src/components/Sidebar.tsx:66-71` — on desktop the per-row delete `×` is hover-gated *and* tiny *and* low-contrast, making it hard to hit precisely with a mouse.** The class is `block text-muted-foreground hover:text-destructive md:hidden md:group-hover:block`. On desktop it only appears on row-hover (correct pattern), but it's a bare 1-character `×` glyph with no padding, no `size-*`, sitting between a flex-grown title and a timestamp — a small, ambiguous mouse target, and on hover the timestamp and `×` occupy the same visual slot (the `×` is appended after the timestamp, not replacing it), so the row reflows/crowds on hover. Desktop has mouse precision but this target fights it. *Fix: give the delete control a real hit area and stable layout — wrap it as a `Button variant="ghost" size="icon-xs"` (existing primitive, `size-6`) using a lucide `X`/`Trash2` icon instead of the raw glyph, and reserve its slot so the row doesn't reflow on hover (e.g. render it always-present with `opacity-0 group-hover:opacity-100` rather than `hidden`/`block`). Reuses existing Button + lucide; no new component.* Confidence: **medium** (classes read directly; "hard to hit / reflow" is inferred from layout, would be obvious in a render).

## Medium Priority (rough edges)

1. **`src/components/Chat.tsx:65 / 74` — the chat column is centered at `max-w-4xl` but the empty-state and composer are also centered, leaving the rest of an ultrawide (2560/3440px) viewport as dead space with the sidebar pinned far left.** This is *mostly* intentional and good (one focused conversation; see Multi-column note below), but the visual result at 3440px is a 288px sidebar hard against the left edge, an 896px column floating in the center, and ~2200px of empty background between them — the sidebar looks "lost" rather than anchored. Not a bug. *Optional, low-effort polish: none required; if it bothers Brian, the cheapest move is to let the chat column sit slightly left-of-center to visually tie it to the sidebar (e.g. cap the centering container) rather than dead-centering in the full remaining width. No new feature, just a margin tweak. I'd leave it.* Confidence: **medium** (layout math is certain; whether it "looks lost" is a judgment call needing a render).

2. **`src/components/ui/dialog.tsx:62` + call sites — dialogs use a fixed `max-h` (`sm:max-h-[80vh]`) with `overflow-y-auto` on the *whole* content, so the sticky header (`DialogTitle`) and close button scroll away with the body in a long Settings/transcript.** The entire `DialogContent` is the scroll container (`overflow-y-auto` is on it at every call site), and the `DialogHeader`/title are inside it — so scrolling a long transcript scrolls the "Task — running" title and the `×` close out of view. On desktop with a mouse wheel this is a minor but real annoyance (you lose the close affordance and context). *Fix: keep the title pinned by making the body (not the whole content) the scroller — move `overflow-y-auto` onto the inner `space-y-*` wrapper and let `DialogContent` be `flex flex-col` with the header `shrink-0`. Standard shadcn scrollable-dialog layout; no new primitive.* Confidence: **medium** (structure read directly; exact scroll feel needs a render).

3. **`src/components/Message.tsx:30-49` — `ToolCallCard` is collapsed by default and the only affordance is a small triangle + monospace tool name; on desktop there's no hover cue that the whole row is clickable.** The button is full-width and clickable (`flex w-full … text-left`), which is good, but it has no `hover:` background, no `cursor` change beyond default button, and no focus-visible ring distinct from its surroundings — a mouse user may not realize the collapsed tool calls expand. *Fix: add `hover:bg-muted/50 rounded-md` to the toggle button (`Message.tsx:33`) so the clickable region is discoverable on hover; it already focuses as a `<button>` but adding the standard `focus-visible:ring` would help keyboard users. Pure utility additions.* Confidence: **medium**.

4. **`src/components/Sidebar.tsx:53-65` — session rows are clickable `<div onClick>`, not buttons, so they're not keyboard-focusable or activatable; a keyboard-only user can't navigate the session list.** The row is `<div onClick={() => onSelect(s.id)}>` with no `role`, `tabIndex`, or key handler. Tab order skips the entire session list — a power user can open Settings/Tasks/New chat (real `<button>`s) but cannot select an existing conversation without the mouse. *Fix: either make the row a `<button>` (and move the delete `×` out of it to avoid nested buttons), or add `role="button" tabIndex={0}` + an `onKeyDown` for Enter/Space. The button route is cleaner and inherits focus-visible automatically.* Confidence: **high** (element type and absence of keyboard handlers read directly).

## Dialog-Specific Concerns (Brian's explicit flag)

- **The root cause is structural, not per-dialog: `DialogContent` is a CSS `grid`, and its content track has `min-width: auto`.** (Verified: `grid w-full max-w-[calc(100%-2rem)] … sm:max-w-sm` in `ui/dialog.tsx:62`; `.grid{display:grid}` and the `max-w-2xl/3xl` utilities all present in compiled CSS.) `max-w-2xl`/`max-w-3xl` cap the dialog *only if every child can shrink to fit*. The fix that addresses the whole class of bugs is `[&>*]:min-w-0` on `DialogContent` (or `min-w-0` on each body wrapper). This is the single most important change in this audit and the literal answer to "dialogs not constraining their content."

- **Settings (`Settings.tsx:56-115`) is actually well-guarded today** — persona `Textarea` carries `w-full min-w-0`; the native `<select>` is `w-full`; schedule `<li>` rows use `min-w-0 flex-1` + `truncate` on label and prompt. At `max-w-2xl` (672px) the 12-row persona textarea and model dropdown are comfortable, not cramped. No content overflow here *given current content*. The risk is latent (a schedule label/prompt is `truncate`, so even a 2000-char prompt won't blow it out). Verdict: fine, but it's only fine because of the per-child guards — the grid `min-w-0` fix (#1) is what makes it *robustly* fine.

- **TasksDialog / TaskCard (`TaskCard.tsx:96-122`)** — task label and `resultSummary` are `truncate` inside `min-w-0 flex-1`; the Cancel/View buttons are `shrink-0`. Long task text wraps/truncates correctly inside `max-w-2xl`. One desktop nit: the action row (`status ● | label | Cancel | View`) doesn't `flex-wrap`, so a very long label competes with two buttons — but `truncate` saves it. Safe.

- **TaskTranscriptDialog (`TaskCard.tsx:66-90`) is the one with live overflow risk**, because it renders arbitrary assistant output via `<Message>` inside `max-w-3xl`. Two sub-cases: (a) **code blocks / tool output** — *safe*: prose `<pre>` compiles to `overflow-x:auto`, and `ToolCallCard`'s arg/result `<pre>`s are `whitespace-pre-wrap` (they wrap, verified `Message.tsx:43,47`). (b) **markdown tables** — *not safe*: `.prose table` is `width:100%; table-layout:auto` with no overflow wrapper (verified in compiled typography CSS), so a wide table expands the grid item and the dialog (Critical #2). Long URLs/inline code are safe thanks to `break-words [overflow-wrap:anywhere]` on the prose div (`Message.tsx:86`). So the transcript dialog's only real blow-out vector is tables — fix via #2.

- **Net:** Brian's instinct is correct, but the symptom he's seeing is almost certainly **markdown tables in the transcript (or chat)**, riding on the **grid `min-width:auto`** structural gap. Fix both (`min-w-0` on the grid + a table overflow wrapper) and the dialog content is constrained by construction, not by luck.

## Strengths

- **The `max-w-4xl` chat constraint (Chat.tsx:65, 74) is the right call and is applied consistently** to both the message list and the composer, so the reading column and input align on wide screens. This was the big desktop win from PR #25 and it holds up.
- **Dialogs already have the responsive full-screen→centered pattern** (`h-[100dvh] … sm:h-auto sm:max-h-[80vh] sm:rounded-xl`) and `overflow-y-auto`, so they scroll instead of overflowing the viewport. Good shadcn hygiene.
- **Prose long-content guards are mostly in place**: `break-words [overflow-wrap:anywhere]` on the markdown container (Message.tsx:86) and `overflow-x:auto` on prose `<pre>` (verified in compiled CSS) mean URLs, inline code, and code blocks are handled — tables are the only gap.
- **Flex-overflow discipline is good** where it exists: `min-w-0 flex-1` + `truncate` on the rows that need it (Sidebar session title, TaskCard label, Settings schedule rows). The grid-item gap (#1) is the one place this discipline didn't reach.
- **`md:contents` on the sidebar wrapper (App.tsx:31) is a tidy way to share one `<Sidebar>` between the permanent desktop layout and the mobile Sheet** without duplicating markup — genuinely clean.
- **Scrollbar placement is correct on desktop**: the chat scroller is the outer `flex-1 overflow-y-auto` (Chat.tsx:62) with the `max-w-4xl` wrapper *inside* it, so the scrollbar sits at the viewport edge, not inside the reading column. This was the #24 fix and it's right.
- **Focus-visible rings are present on every interactive primitive** (button, input, textarea, select-trigger all carry `focus-visible:ring-3 ring-ring/50`) — keyboard focus is *styled*; the gap is only that the sidebar session rows aren't focusable elements (Medium #4).

## Suggested Next Steps

1. **Close the grid `min-width:auto` gap (Critical #1)** — add `[&>*]:min-w-0` to `DialogContent` in `ui/dialog.tsx:62`. One line; makes `max-w` authoritative for all three dialogs and is the direct answer to Brian's flag.
2. **Wrap prose tables in an overflow container (Critical #2)** — add `[&_table]:block [&_table]:overflow-x-auto [&_table]:max-w-full` (or `overflow-x-auto` on the prose div) at `Message.tsx:86`. Fixes the actual blow-out symptom in both the transcript dialog and the chat column.
3. **Scope the coarse-pointer rule (High #1)** — change `@media (pointer: coarse)` → `@media (pointer: coarse) and (hover: none)` in `index.css` so touchscreen laptops keep desktop density. One token.
4. **Guard the composer against IME, optionally add Cmd/Ctrl+Enter (High #2)** — add `!e.nativeEvent.isComposing` to the Enter-send branch in `Chat.tsx:78`; correct regardless of the Cmd+Enter product decision.
5. **Make session rows keyboard-navigable (Medium #4)** and give the delete control a real hit area (High #3) — convert the Sidebar row to a `<button>` (or `role/tabIndex/onKeyDown`) and swap the raw `×` for `Button size="icon-xs"` + lucide icon with a reserved slot. Then smoke-test the table-overflow case at 1440/1920/2560 to confirm #1+#2.
