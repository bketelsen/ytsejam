# Desktop UI Audit — GPT-5.5

## Verdict
NEEDS-WORK — The desktop shell is mostly sound, but the dialog max-width overrides are not actually taking effect at `sm` and up, and keyboard navigation in the sidebar is incomplete.

## Critical Issues (broken or significantly degraded desktop UX)
1. `src/components/ui/dialog.tsx:62` + `src/components/Settings.tsx:56`, `src/components/TasksDialog.tsx:20`, `src/components/TaskCard.tsx:69` — The app dialogs pass `max-w-2xl` / `max-w-3xl`, but the primitive also contributes `sm:max-w-sm`; because the caller only overrides the unprefixed max width, desktop widths still match the primitive’s `sm:max-w-sm` cap. That makes Settings, Tasks, and transcript dialogs much narrower than their source code appears to intend, especially for a 12-row persona editor and transcript content. Suggested fix: override the same responsive variant at call sites, e.g. `sm:max-w-2xl` for Settings/Tasks and `sm:max-w-3xl` for TaskTranscriptDialog, while keeping the mobile/full-height classes. Alternatively change the primitive default to a less opinionated desktop max-width and let callers choose. Confidence: high.

## High Priority (degrades desktop ergonomics)
1. `src/components/Sidebar.tsx:54-57` — Session rows are clickable `<div>` elements, not buttons/links, so they are not reachable or activatable via Tab/Enter/Space. A keyboard-first desktop user can tab to New chat/Tasks/Settings and the delete controls, but cannot select a conversation without a mouse. Suggested fix: render each session as a `<button type="button">` with `w-full text-left` and the existing hover/active styling, or add `tabIndex={0}`, `role="button"`, and Enter/Space handling if a real button is impractical. Prefer the real button pattern. Confidence: high.
2. `src/components/Sidebar.tsx:66-70` — The delete control is hover-revealed at `md+` (`md:hidden md:group-hover:block`) and is a raw `<button>` with no shadcn focus-visible ring. Because the parent row is not focusable and there is no `group-focus-within:block`, keyboard users may tab to an invisible control or get no clear focus indication. Suggested fix: use the existing `<Button variant="ghost" size="icon-sm">` pattern with `aria-label`, and reveal it with `md:group-hover:block md:group-focus-within:block` or keep it visible when focused. Confidence: high.
3. `src/components/TaskCard.tsx:69-78` + `src/components/Message.tsx:72-86` — The transcript dialog renders normal chat bubbles inside a dialog; assistant/user bubbles are capped at `max-w-[80%]`, so even after the dialog is widened the effective transcript reading width is only 80% of the dialog. For desktop transcript review, this wastes space and makes code/tool-heavy messages more cramped than the surrounding `max-w-3xl` suggests. Suggested fix: allow a transcript/dialog mode for `Message` that uses `max-w-full` or `max-w-[min(100%,...)]`, or wrap transcript messages with a context class and override the bubble width there. Confidence: medium.

## Medium Priority (rough edges)
1. `src/components/TaskCard.tsx:71-72` — The transcript dialog title directly interpolates `task.label` with no truncation, wrapping, or right padding for the absolute close button from `dialog.tsx:70-73`. A long task label, URL, or generated title can collide with the close button or create awkward header wrapping in a constrained dialog. Suggested fix: add header/title constraints such as `DialogHeader className="pr-8"` and `DialogTitle className="break-words leading-snug"`, or truncate the label with the full label exposed via `title`. Confidence: medium.
2. `src/components/Message.tsx:86-87` — Markdown is rendered in a `prose` container with `max-w-none break-words [overflow-wrap:anywhere]`, but there are no explicit table/pre constraints at the component level. Tailwind Typography often handles code block overflow, but wide GFM tables and unusual pre/code output should be explicitly protected in this app because dialogs and chat both reuse `Message`. Suggested fix: add targeted typography utilities such as `prose-pre:max-w-full prose-pre:overflow-x-auto prose-table:block prose-table:max-w-full prose-table:overflow-x-auto` or a small markdown-content wrapper using existing Tailwind classes. Confidence: medium.
3. `src/components/Chat.tsx:77-81` — The composer sends on bare Enter and supports Shift+Enter for newline, but it does not support Cmd/Ctrl+Enter as an additional desktop shortcut. Since this is a multiline textarea, power users coming from desktop chat/tools may expect Cmd/Ctrl+Enter to send. Suggested fix: accept either bare Enter without Shift as today or `(e.metaKey || e.ctrlKey) && e.key === "Enter"`; optionally expose `aria-keyshortcuts="Enter Control+Enter Meta+Enter"`. Confidence: medium.
4. `src/components/Sidebar.tsx:48-50` — The Settings button is rendered as a bare `⚙` glyph without an explicit accessible name. It is mouse-discoverable enough, but keyboard/screen-reader ergonomics are weaker than the Tasks/New chat buttons. Suggested fix: replace with a lucide icon plus `<span className="sr-only">Settings</span>` or add `aria-label="Settings"`. Confidence: medium.

## Dialog-Specific Concerns (Brian's explicit flag)
- Settings (`src/components/Settings.tsx:56-66`): The intended `max-w-2xl` desktop box is likely defeated by the primitive’s `sm:max-w-sm` default (`src/components/ui/dialog.tsx:62`). In a true 2xl box, the 12-row persona textarea and model select should be acceptable; in the effective sm box, they will feel cramped and the schedule cards have less room for label/status/prompt text.
- Settings schedules (`src/components/Settings.tsx:99-112`): Schedule cards correctly use `min-w-0`, `flex-1`, and `truncate` for label/prompt, so normal long text should not push the dialog wider. The secondary status span is not constrained, but its content appears bounded by cron/locale-date text, so this is a lower-risk issue.
- TasksDialog (`src/components/TasksDialog.tsx:20-28` + `src/components/TaskCard.tsx:101-122`): Task rows are structurally protected with `min-w-0`, truncation, and fixed action buttons, so long task labels/summaries should not expand the dialog. The main problem is again the dialog width override: `max-w-2xl` should become `sm:max-w-2xl` to beat the primitive default.
- Task transcript dialog (`src/components/TaskCard.tsx:69-79`): The intended `max-w-3xl` is likely not applied at desktop because of `sm:max-w-sm`. This is the highest-impact dialog case: transcripts render full `<Message>` content including markdown and tool-call cards, so the accidental narrow width amplifies wrapping and nested scrolling.
- Task transcript title (`src/components/TaskCard.tsx:71-72`): Long task labels are unconstrained in the dialog header and can conflict with the absolute close button. Add right padding and wrapping/truncation.
- Tool-call output (`src/components/Message.tsx:41-48`): Expanded tool-call details are reasonably constrained with `max-h-64 overflow-auto` and `whitespace-pre-wrap`; they should not push the dialog horizontally in typical cases. This creates intentional nested scrolling inside transcript dialogs, which should be checked in-browser for mouse-wheel behavior.

## What I Could NOT Verify Without a Browser
- Actual rendered dialog widths at 1440/1920/2560/3440px and whether the `sm:max-w-sm` cascade manifests exactly as predicted in the runtime class string.
- Whether GFM tables, long code fences, and very long URLs create horizontal page/dialog overflow after Tailwind Typography’s generated CSS is applied.
- Visual balance of the fixed `w-72` sidebar next to the centered `max-w-4xl` chat column on ultrawide monitors.
- Exact scrollbar placement and mouse-wheel chaining in nested contexts: main chat scroll area, sidebar scroll area, dialog scroll area, and expanded tool-result scroll areas.
- Focus-ring visibility in the final rendered theme, especially for raw native controls and the sidebar delete button.
- Resize feel around the `md` breakpoint while dragging between tablet-ish and desktop widths.

## Strengths
- The main chat scroll container is outside the `max-w-4xl` reading wrapper (`src/components/Chat.tsx:60-61`), so the main scrollbar should sit at the viewport/chat-pane edge rather than inside the reading column.
- The desktop layout is intentionally simple: permanent `w-72` sidebar at `md+` and a centered, readable chat column. That looks like a coherent “one focused conversation” product call rather than an accidental underuse of space.
- Sidebar and dialog lists generally use `min-w-0`, `truncate`, and fixed action buttons where needed, which is the right Tailwind pattern for preventing long labels from widening cards.
- Desktop pointer density is preserved: button sizes are compact by default (`src/components/ui/button.tsx:23-34`), and the 44px target bump is scoped only to coarse pointers (`src/index.css:150-155`).
- Radix dialogs/sheets should provide baseline Escape-to-close and focus trapping behavior through the existing shadcn primitives.
- Empty states are present and intentional: chat (`src/components/Chat.tsx:62-64`), sessions (`src/components/Sidebar.tsx:78`), schedules (`src/components/Settings.tsx:94-95`), tasks (`src/components/TasksDialog.tsx:25`), and transcript (`src/components/TaskCard.tsx:76`).

## Suggested Next Steps
1. Fix dialog sizing first: change the three dialog call-site classes to use responsive overrides (`sm:max-w-2xl` / `sm:max-w-3xl`) and inspect at 1440px and 2560px.
2. Convert sidebar session rows to real keyboard-operable buttons and make the delete affordance visible on focus as well as hover.
3. Add transcript-specific message width behavior so transcript dialogs can use their full desktop width.
4. Harden markdown overflow with explicit `prose-pre` / `prose-table` constraints, then test with a long code block and a wide GFM table.
5. Do a short browser pass at 1440, 1920, 2560, and 3440px focusing on dialog width, scrollbars, focus order, and ultrawide visual balance.
