# Desktop UI Audit — Gemini 3.5 Flash

## Verdict
NEEDS-WORK — While the core chat layout reflows nicely, critical keyboard navigation blocks and dialog viewport squishing significantly degrade the desktop ergonomics.

## Critical Issues (broken or significantly degraded desktop UX)

1. **file: `web/src/components/Sidebar.tsx` (lines 53-75)**
   - **Problem:** Chat session selection items are rendered as flat `div` blocks with an `onClick` handler but no keyboard interactivity whatsoever (no `tabIndex`, no ARIA roles, and no `onKeyDown` handlers). Power users driving the app exclusively via keyboard have zero ability to select, switch, or tab to existing chat sessions.
   - **Suggested Fix:** Refactor the session rows to include `role="button"`, `tabIndex={0}`, and an `onKeyDown` handler that intercepts `Enter` or `Space` keys to call `onSelect(s.id)`. Apply focus ring styling: `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-sidebar-ring`.
   - **Confidence:** High (Direct code inspection)

2. **file: `web/src/components/Sidebar.tsx` (line 69)**
   - **Problem:** The session delete button is a semantic `<button>` but is hidden on desktop via `md:hidden md:group-hover:block`. While invisible, it is still focusable in the tab sequence. A keyboard user will focus a completely invisible button; hitting `Enter` or `Space` will trigger `confirm("Delete this session?")` without any visual indication of focus, causing a massive UX and accessibility hazard.
   - **Suggested Fix:** Ensure the delete button is revealed when its parent session row is focused by the keyboard. Update the utility classes to: `md:hidden md:group-hover:block md:group-focus-within:block focus-visible:block focus-visible:ring-2 focus-visible:ring-destructive focus-visible:outline-none rounded`.
   - **Confidence:** High (Direct code inspection)

## High Priority (degrades desktop ergonomics)

1. **files: `web/src/components/Settings.tsx` (line 56), `web/src/components/TasksDialog.tsx` (line 20), `web/src/components/TaskCard.tsx` (line 69)**
   - **Problem:** Custom dialog maximum widths on desktop (such as `max-w-2xl` and `max-w-3xl`) are overridden and flattened to a narrow `max-w-sm` (384px) because of the hardcoded `sm:max-w-sm` class in the primitive `DialogContent` (`web/src/components/ui/dialog.tsx` line 62). Since `twMerge` treats responsive modifiers independently from unprefixed ones, unprefixed classes cannot override media-query specific classes, squeezing complex dialog views (schedules lists, task cards, and transcripts) into a cramped mobile-sized viewport on desktop.
   - **Suggested Fix:** Explicitly pass the responsive prefix modifier in the dialog instances:
     - `Settings.tsx`: use `sm:max-w-2xl`
     - `TasksDialog.tsx`: use `sm:max-w-2xl`
     - `TaskCard.tsx` (TaskTranscriptDialog): use `sm:max-w-3xl`
   - **Confidence:** High (Tailwind merging mechanics)

2. **files: `web/src/components/Settings.tsx` (line 56), `web/src/components/TasksDialog.tsx` (line 20), `web/src/components/TaskCard.tsx` (line 69)**
   - **Problem:** Direct application of `overflow-y-auto` to `DialogContent` causes the entire modal content (including the header, title, and the absolute-positioned "X" close button) to scroll. On long forms or transcripts, the close button and context header scroll out of view.
   - **Suggested Fix:** Implement structured dialog scrolling. Remove `overflow-y-auto` from the `DialogContent` props, add `flex flex-col overflow-hidden p-0`, insert a bordered header, and wrap dialog children in an inner scroll container `<div className="flex-1 overflow-y-auto p-4 space-y-4">`. This pins the close button and title to the top while only body content scrolls.
   - **Confidence:** High (Layout design)

## Medium Priority (rough edges)

1. **file: `web/src/components/Message.tsx` (lines 74-87) & `web/src/index.css`**
   - **Problem:** Markdown tables generated by the assistant inside the prose container do not have overflow boundaries. If the model outputs a wide multi-column table, it can blowout the message bubble (`max-w-[80%]`), causing clipping or breaking the main viewport layout.
   - **Suggested Fix:**
     - Add `.prose table { display: block; width: 100%; overflow-x: auto; }` to `web/src/index.css` under the `@layer base` directive.
     - Add `min-w-0` to the message bubble `div` in `Message.tsx` (line 74): `className="max-w-[80%] min-w-0 rounded-lg..."` to allow the flex element to shrink and trigger the horizontal scroll bar inside the code block/table.
   - **Confidence:** High (CSS flexbox specs)

2. **file: `web/src/components/Message.tsx` (lines 33-35)**
   - **Problem:** The tool call toggle button (`ToolCallCard`) does not have any hover background or feedback transition class. It is non-obvious to mouse users that this card is an interactive, clickable accordion.
   - **Suggested Fix:** Add visual hover feedback such as `hover:bg-muted/50 transition-colors` to the button.
   - **Confidence:** High (Visual affordance audit)

## Dialog-Specific Concerns (Brian's explicit flag)

- **Settings Dialog (`Settings.tsx`):** The 12-row persona `<Textarea>` and schedules list feel extremely squeezed horizontally. Due to the primitive's `sm:max-w-sm` taking precedence over the unprefixed `max-w-2xl` on desktop, the text container is only 384px wide, wrapping lines aggressively and rendering the text area highly claustrophobic.
- **Tasks Dialog (`TasksDialog.tsx`):** Task rows have a three-column split layout (Status indicator, task description, cancel/view button). Squeezing this structure into a 384px column causes extreme text truncation for task labels and results summaries, making them unreadable.
- **Task Transcript Dialog (`TaskCard.tsx`):** Transcripts render full AI tool call chains and assistant messages containing code blocks. Restricting code blocks and complex tool outputs to a narrow mobile column forces heavy vertical stretching, excessive wrapping, and breaks code legibility.

## What I Could NOT Verify Without a Browser

1. **Focus Ring Contrast and Aesthetics:** We verified that `focus-visible:ring-3` rules are present on inputs and buttons, but actual visual contrast, visibility in dark mode, and ring positioning on smaller components (such as the toggle accordions) need live screen inspection.
2. **Scrollbar Aesthetics:** While the scrollbar correctly attaches to the viewport edge instead of the centered chat container, its default visual style (system scrollbars vs. custom hidden/thin track scrollbars) cannot be evaluated without rendering.
3. **Ultrawide Empty Space Visual Weight:** On viewports exceeding 2560px, the 288px sidebar on the left and the 896px centered conversation column are separated by large swaths of solid background. Whether this empty space looks clean or awkwardly sparse requires visual rendering verification.

## Strengths

- **Coarse Pointer Target Isolation:** Touch target expansions (to 44px) are brilliantly restricted to `@media (pointer: coarse)` in `index.css`. Desktop density is fully preserved for fine pointer mouse users.
- **Layout Reflow Performance:** The transition between a drawer-based sidebar on mobile and a fixed structural sibling sidebar on desktop is handled highly efficiently via the `md:contents` display property wrapper in `App.tsx` (preventing flex-nesting wrapper pollution).
- **Desktop Keyboard Composer Shortcuts:** Message submission supports standard desktop expectations natively (Enter to submit, Shift+Enter for newlines in `Chat.tsx`).
- **PWA Integration:** Home screen install parameters and manifest files are correctly integrated, which improves windowing options on desktop.

## Suggested Next Steps

1. **Resolve Dialog Max-Width Constraints:** Replace unprefixed dialog size overrides with media-query responsive prefixes (`sm:max-w-2xl` and `sm:max-w-3xl`) across `Settings.tsx`, `TasksDialog.tsx`, and `TaskCard.tsx` so they correctly widen on desktop screens.
2. **Implement Structural Dialog Scrolling:** Refactor dialog content structure by using `p-0 flex flex-col overflow-hidden` on `DialogContent` and placing scrolling wrapper logic exclusively on the body elements, pinning the header and close buttons.
3. **Establish Sidebar Keyboard Interactivity:** Add keyboard focus support (`tabIndex`, role, keydown event handlers) to structural sidebar list elements and sync the delete button visibility with focus states (`group-focus-within`).
4. **Prevent Markdown Table Blowout:** Inject table-wrapping CSS rules and message bubble `min-w-0` styles to make sure complex markdown tables do not overflow.
